# Renderer 启动诊断与错误弹窗设计

## 1. 目标

当 `blackhole.exe` 缺失、无法启动、启动后立即退出、初始化失败或长时间未进入可用状态时，Qt 主界面必须向用户显示可理解、可复制、可用于反馈的错误信息。软件在托盘或后台时发生故障，应自动恢复主窗口并置前。

本功能只处理 Renderer 启动和早期运行故障，不改变正常停止、空闲检测、媒体检测和游戏检测策略。

## 2. 现状与问题

- `BlackHoleCore::startRenderer()` 已能通过 `QProcess::errorOccurred` 报告进程启动错误，但 QML 没有显示该信号。
- `QProcess::finished` 不区分用户主动停止与启动阶段异常退出。
- Renderer 的详细原因主要写入 `blackhole_debug.txt`，普通用户不知道日志位置，也无法判断该提供哪些信息。
- 进程存在不代表渲染成功；窗口、捕获、OpenGL、Shader 等初始化失败可能发生在进程启动之后。
- 空闲检测模式可能重复尝试启动，若每次都弹窗会形成错误刷屏。

## 3. 方案选择

采用“启动前文件检查 + 进程事件 + 日志就绪握手 + 超时诊断”的组合方案。

不单独依赖 `QProcess::errorOccurred`，因为它只能覆盖操作系统拒绝创建进程的情况；也不单独依赖退出码，因为进程可能保持运行但未成功显示黑洞。

## 4. 启动状态模型

Qt 后端维护以下状态：

- `Stopped`：Renderer 未运行。
- `Starting`：进程已请求启动，等待 Renderer 就绪。
- `Ready`：日志已出现明确的就绪标记，Renderer 可用。
- `Stopping`：用户或程序正常请求停止，不应弹出故障。
- `Failed`：启动前检查、进程创建、早期退出、日志失败标记或就绪超时。

每次启动生成一个内部启动序号。定时器、进程事件和日志检查只处理当前序号，防止旧回调误报新进程。

## 5. 启动前检查

启动前至少检查：

- `blackhole.exe` 存在且为普通文件。
- Renderer 工作目录存在。
- `blackhole.glsl` 存在。
- `shaders/vert.glsl` 存在。
- `shaders/frag_desktop_header.glsl` 存在。

不维护一份容易过时的全部 Qt DLL 清单。Windows 加载器缺少 Renderer 依赖 DLL 时，由 `QProcess::FailedToStart` 和系统错误字符串报告；发布脚本继续负责完整 Release 文件校验。

## 6. 就绪握手和故障判定

Renderer 继续使用 UTF-8 的 `blackhole_debug.txt` 作为诊断通道，并写入稳定标记：

- 成功：`[OK] Ready, entering main loop`
- 失败：现有 `[FAIL] ...` 标记

Qt 在进程启动后轮询当前工作目录的日志：

- 发现就绪标记：状态切换为 `Ready`，停止启动诊断定时器。
- 发现 `[FAIL]`：立即进入 `Failed`，提取最后一条失败及日志尾部。
- 进程在就绪前退出：进入 `Failed`，记录退出码和退出状态。
- 8 秒内没有就绪标记：进入 `Failed`，提示初始化超时，并附带当前日志尾部。

启动前记录日志文件原始大小和修改时间，避免把上一次运行的就绪标记误认为本次成功。若 Renderer 启动时截断日志，则从新文件起始读取；若未截断，只读取本次启动后新增内容。

## 7. 错误详情模型

后端向 QML 暴露一次性的结构化错误信息：

- 标题：例如“黑洞渲染器启动失败”。
- 用户说明：简短说明失败阶段和建议动作。
- 技术详情：错误分类、退出码、进程错误、缺失文件、日志尾部。
- 日志路径：使用绝对路径。
- 唯一故障标识：由启动序号和错误类别组成。

错误类别至少包含：

- `MissingFile`
- `FailedToStart`
- `ExitedBeforeReady`
- `InitializationFailed`
- `ReadyTimeout`

日志尾部限制长度，避免超大日志阻塞 UI；保留最后约 8 KiB，并以 UTF-8 容错读取。

## 8. 弹窗与窗口行为

新增专用 Renderer 错误对话框，沿用现有主题和无边框界面风格。弹窗包含：

- 简明错误原因。
- 可选择和复制的技术详情文本。
- “复制详情”按钮。
- “打开日志目录”按钮；没有日志文件时打开预期目录。
- “关闭”按钮。

收到新故障时：

1. 若主窗口隐藏或最小化，则恢复并显示。
2. 让主窗口置前并请求激活。
3. 打开错误弹窗。

系统托盘状态同步为主窗口已显示。

## 9. 防重复与正常停止

- `stopRenderer()` 在终止进程前进入 `Stopping`，随后产生的 `finished` 不视为错误。
- 已进入 `Ready` 后的正常退出沿用现有停止行为；本功能不把所有运行期退出都解释为启动失败。
- 同一启动序号最多弹出一次。
- 空闲检测在同一故障未被用户关闭或处理前不重复弹窗。
- 下一次明确启动会生成新序号，可再次报告新的故障。

## 10. 测试策略

新增独立、可测试的诊断状态模块，避免测试真实 GPU 和窗口：

- 缺少 Renderer 或 Shader 文件时生成 `MissingFile`。
- `QProcess::FailedToStart` 生成 `FailedToStart`。
- 就绪前退出生成 `ExitedBeforeReady`，包含退出码。
- 新日志包含 `[FAIL]` 时生成 `InitializationFailed`。
- 超过 8 秒未就绪生成 `ReadyTimeout`。
- 新日志包含 Ready 标记时进入 `Ready`，不生成错误。
- `Stopping` 状态下退出不生成错误。
- 同一启动序号不重复生成故障。
- 旧日志中的 Ready 标记不算本次启动成功。

QML 契约检查覆盖弹窗入口、复制详情、打开日志目录以及隐藏窗口恢复逻辑。最后执行 Qt Release 构建、现有 CTest、Renderer 实际启动和 Release 打包验证。

## 11. 文件边界

预计修改：

- 新增 Renderer 启动诊断状态模块及测试。
- 修改 `BlackHoleCore`，接入文件检查、QProcess 事件、日志轮询和结构化错误信号。
- 新增 Renderer 错误对话框 QML，并在 `Main.qml` 接收错误、恢复窗口和打开弹窗。
- 修改 QML 资源和 CMake 测试目标。
- 增加静态契约检查。

不修改 Renderer 图形算法、空闲检测判定规则、媒体/游戏识别规则和更新系统。

## 12. 验收标准

- 删除任一必需 Renderer 文件后点击启动，会立即出现明确缺失文件弹窗。
- Renderer 无法由 Windows 创建时，弹窗显示系统错误信息。
- Renderer 在 Ready 前退出或日志输出 `[FAIL]` 时，8 秒内出现弹窗。
- Renderer 卡在初始化阶段时，约 8 秒后出现超时弹窗。
- 托盘或隐藏状态发生故障时，主窗口恢复并置前。
- 技术详情可复制，日志目录可打开。
- 用户主动停止不弹错误，同一次故障不重复弹出。
- 正常 Renderer 启动不出现错误弹窗。
