# 开机自启动与媒体/游戏检测增强设计

日期：2026-07-12

## 1. 背景与目标

当前 Qt 界面存在三类相互关联但应独立处理的问题：

1. 关闭“开机自动启动”后，注册表项仍然存在。
2. 视频和游戏检测依赖固定进程名单、窗口大小与音频峰值，哔哩哔哩 Windows 客户端以及窗口化游戏容易漏检。
3. Windows 桌面窗口可能被“覆盖全屏且非最大化”规则误判为无边框全屏程序，导致用户停留桌面时黑洞不触发。

本阶段目标是建立适合公开发布的低开销检测体系：

- 开机自启动开关立即生效，界面状态与注册表状态一致。
- 视频仅在实际播放时阻止黑洞，暂停后允许按空闲时间触发。
- 覆盖常用国内视频客户端与浏览器播放场景，重点验证哔哩哔哩 Windows 客户端。
- 游戏自动检测覆盖独占全屏、无边框全屏和一部分窗口化游戏。
- 自动游戏检测采用保守策略，允许少量漏检，避免大量普通软件被误判。
- 用户可通过前台强制名单补充漏检应用。
- 修复桌面空闲时不触发的问题，并提供可观察的判定原因。

## 2. 非目标

- 不承诺在完全不读取平台信息、没有媒体会话、没有图形特征且不在用户名单中的情况下识别所有窗口化游戏。
- 不使用进程注入、驱动、DLL 注入或需要管理员权限的检测方式。
- 不把 CPU/GPU 功耗作为唯一游戏依据；低功耗游戏必须可以通过其他信号或用户名单覆盖。
- 不因后台常驻的播放器、Steam 或游戏启动器永久阻止黑洞。

## 3. 方案选择

### 3.1 选定方案：多信号检测

采用 Windows 系统媒体会话、前台窗口信息、音频会话、Windows 游戏记录、游戏平台安装信息和用户配置名单组合判定。

选择原因：

- 比继续扩充硬编码进程名更能适应客户端更新。
- 比单纯检测音频更能区分播放与暂停，并覆盖无声但上报媒体状态的视频。
- 比仅检测功耗更能覆盖低功耗游戏。
- 无需侵入其他进程，适合公开发布。

### 3.2 未选方案

- 仅扩充进程名单：实现简单，但版本变化后继续漏检，窗口化游戏覆盖不足。
- ETW、模块深扫或进程注入：兼容性和维护成本高，可能增加耗电或触发安全软件。

## 4. 总体架构

新增独立的前台活动检测模块，Qt 监控与原生监控共用相同判定规则，避免继续维护两份逐渐分叉的实现。

模块输入：

- 当前前台窗口句柄、窗口类、样式、矩形、进程 ID 与可执行文件路径。
- Windows 系统媒体会话的播放状态与来源应用。
- 当前音频会话及其进程信息。
- 用户配置的忽略名单、媒体提示名单和前台强制名单。
- Windows 游戏识别记录和已支持游戏平台的安装库信息。

模块输出统一为结构化结果：

- 是否阻止黑洞触发。
- 判定类别：桌面、普通应用、媒体播放、全屏游戏、窗口化游戏、强制名单或检测失败。
- 判定原因与命中的进程标识。

Qt `checkIdle()` 与原生 `isWatchingVideo()` 不再分别复制检测细节，只消费统一结果。

## 5. 判定顺序

检测按以下优先级执行，前面的明确结果可以提前返回：

1. 排除黑洞自身窗口。
2. 识别并排除 Windows 桌面和系统外壳窗口。
3. 命中“始终允许触发名单”时不阻止。
4. 前台进程命中“前台强制不触发名单”时立即阻止。
5. 存在与前台应用相关且状态为 `Playing` 的系统媒体会话时阻止。
6. 对不支持系统媒体会话的已知媒体应用，使用前台进程族与音频会话匹配兜底。
7. 检测独占全屏或真正的无边框全屏游戏。
8. 使用 Windows 游戏记录、游戏平台安装库和保守应用特征识别窗口化游戏。
9. 没有可靠信号时不阻止，交由用户强制名单补充。

该顺序保证桌面窗口不会先被全屏规则误判，也保证强制名单不受音频阈值影响。

## 6. 视频检测设计

### 6.1 系统媒体会话优先

读取 Windows 全局系统媒体传输控制会话，只有播放状态为 `Playing` 才视为正在播放：

- `Playing`：阻止黑洞。
- `Paused`、`Stopped`、`Closed`：允许黑洞按空闲时间触发。

媒体会话必须与当前前台应用关联。后台音乐、后台网页或壁纸音频不能仅因存在媒体会话就阻止黑洞。

### 6.2 音频会话回退

部分国内客户端可能不发布系统媒体会话。此时仅对媒体提示名单中的前台应用进行音频检测：

- 按同一可执行文件、同一安装目录或明确的父子进程关系匹配进程族。
- 不再只比较前台进程名与音频会话进程名是否完全相同。
- 音频峰值只作为回退证据，不影响前台强制名单。

哔哩哔哩网页维持现有可用能力并作为回归测试；哔哩哔哩 Windows 客户端作为重点运行测试对象。

### 6.3 国内媒体兼容方式

默认媒体提示名单覆盖常用国内客户端和浏览器，但名单只决定“是否进入媒体回退检测”，不代表始终阻止。后续新增客户端通常只需更新数据名单，不改判定代码。

## 7. 游戏检测设计

### 7.1 全屏与无边框全屏

- 保留 Windows 独占全屏状态检测。
- 无边框全屏必须基于前台窗口所在显示器的边界判断，不能只使用主屏分辨率。
- 在全屏判断前排除 `Progman`、`WorkerW`、任务栏、开始菜单、任务切换器和黑洞自身窗口。

### 7.2 窗口化游戏

采用保守的多信号组合：

- Windows 已记录的游戏应用信息。
- Steam、Epic、GOG、Xbox 等已支持平台的本地安装库与运行程序路径对应关系。
- 可执行文件版本信息、产品信息和明确的游戏运行特征。
- 用户前台强制名单。

单一高 CPU/GPU 占用、单一图形 API 模块或单一窗口样式不足以判定游戏，避免浏览器、Qt 软件和开发工具误判。

### 7.3 低功耗游戏

平台安装库、Windows 游戏记录和用户强制名单不依赖实时功耗，因此可以覆盖低功耗游戏。无法获得这些信号的未知窗口化游戏允许漏检。

## 8. 名单模型与迁移

名单分为三类：

1. 始终允许触发名单：前台命中时忽略媒体与游戏检测。
2. 媒体识别提示名单：前台命中时允许进入媒体会话或音频回退检测，但不是始终阻止。
3. 前台强制不触发名单：前台命中时无条件阻止黑洞；后台运行不生效。

迁移规则：

- 现有 `idleWhitelist` 迁移为“始终允许触发名单”。
- 现有 `idleBlacklist` 默认包含浏览器、播放器和启动器，迁移为“媒体识别提示名单”。
- 新增的“前台强制不触发名单”默认为空。
- 配置文件升级后写入新版本格式；旧格式仍可读取，避免用户配置丢失。

## 9. 名单交互设计

三种添加方式同时保留：

- 从正在运行的程序中选择：显示图标、应用名称、进程名和路径；默认隐藏系统进程，可手动刷新。
- 选择程序文件：使用系统文件选择器选择 `.exe`，适合尚未运行或便携版程序。
- 手动输入：作为高级入口保留，支持输入进程名。

保存时统一规范化：去除多余引号、统一大小写比较、保留展示名称、拒绝空值与重复项。无法读取图标或版本信息时仍允许按进程名添加，并显示明确的降级状态。

## 10. 开机自启动设计

- `autoStart` 属性发生用户切换时立即同步注册表，而不是等待全量配置保存。
- 启用时写入当前 Qt 程序的带引号绝对路径。
- 禁用时同时删除当前值 `BlakholeUI` 和历史值 `BlackholeScreensaver`。
- 写入后调用同步并回读验证；失败时恢复界面状态并显示错误，不假装成功。
- 普通配置保存仍持久化 `autoStart` 字段，但不再承担唯一的注册表同步职责。
- 启动加载时以实际注册表状态校准界面，处理配置文件与注册表不一致。

## 11. 桌面空闲修复

在任何全屏检测之前识别系统外壳窗口：

- `Progman`、`WorkerW` 视为 Windows 桌面，不阻止黑洞。
- `Shell_TrayWnd`、开始菜单、任务切换器等系统 UI 不作为游戏或视频。
- 壁纸音频不会因为后台存在音频会话而阻止，除非对应应用同时位于前台并满足明确媒体条件。

此修复同时应用于 Qt 和原生监控路径。

## 12. 状态与诊断

Qt 界面显示最近一次检测摘要：

- 当前前台应用、进程名和窗口模式。
- 当前是否阻止黑洞。
- 判定原因，例如“媒体会话正在播放”“前台强制名单”“无边框全屏游戏”“Windows 桌面”或“未发现可靠信号”。

诊断信息使用低频更新，不记录窗口标题中的隐私内容，不持续写入高频日志。

## 13. 错误处理与性能

- 媒体会话、音频设备、注册表或进程信息读取失败时返回可诊断状态，不崩溃、不阻塞主线程。
- 常规前台和空闲检查维持约 1 秒周期。
- 平台安装库与 Windows 游戏记录使用缓存，只在启动、手动刷新或目录变化后重建。
- 图标读取和运行程序枚举按用户打开选择器时执行，不在每秒检测路径中运行。
- 所有句柄、COM/WinRT 对象和音频接口必须按所有返回路径释放。

## 14. 分阶段实施

预计修改超过 3 个文件，必须分阶段完成并逐阶段验证：

1. 修复开机自启动即时同步与桌面窗口误判。
2. 抽取共享检测核心并接入媒体会话、音频回退。
3. 接入保守游戏多信号检测与三类名单迁移。
4. 实现三种名单添加入口和诊断状态界面。
5. 完成静态检查、单元/集成测试、Qt 与主程序 Release 构建、运行测试和打包。

## 15. 验收标准

- 关闭开机自启动后无需退出界面，注册表值立即消失；重新打开程序仍显示关闭。
- 停留 Windows 桌面达到空闲阈值后黑洞可以触发，壁纸音频不影响结果。
- 哔哩哔哩网页播放时阻止、暂停时允许触发。
- 哔哩哔哩 Windows 客户端播放时阻止、暂停时允许触发。
- 独占全屏和无边框全屏游戏能够阻止黑洞。
- 至少使用一个平台识别的窗口化游戏验证自动检测，并使用一个未知程序验证不会被激进误判。
- 前台强制名单中的程序无论有声或无声都阻止，切到后台后不再阻止。
- 三种名单添加方式均可用，重复项不会重复保存。
- Qt UI 与原生监控对相同前台状态给出一致判定。
- 主程序和 Qt UI Release 构建无新增错误或警告。

## 16. 已知限制

Windows 没有公开且统一的“任意进程是否为游戏”接口。对于不在平台库、不在 Windows 游戏记录、没有可靠特征且未加入用户名单的窗口化游戏，本设计明确允许漏检。该限制通过易用的前台强制名单和诊断状态降低用户处理成本，而不是采用高误判或高侵入方案。
