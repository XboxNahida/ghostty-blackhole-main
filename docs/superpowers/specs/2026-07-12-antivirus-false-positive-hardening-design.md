# 杀毒软件误报缓解与发布安全加固设计

## 目标

在不破坏黑洞渲染、屏幕捕获、多显示器、空闲检测、自启动和快捷键功能的前提下，减少 `blackhole.exe` 与远控木马启发式规则的行为重叠，提高 Windows 对发布文件的身份识别和信誉积累能力。

本设计目标是“尽量降低误报概率”，不承诺所有杀毒引擎永不报警。

## 已确认事实

- 用户旧版反馈包含 Windows 11 智能应用控制信誉拦截，以及 `Trojan.Win.XWorm.5e082ae8`、`Trojan:Win64/XWorm.GHM!MTB` 等通用 XWorm 检测。
- 当前两个 EXE 均未签名，且缺少产品名、开发者、版权、文件版本和原始文件名资源。
- 当前 PE 节区正常，没有 UPX 或常见压缩壳特征，并已启用 ASLR、高熵地址和 DEP/NX。
- 当前源码和导入表没有网络通信、远程下载、进程注入、凭据读取或由网络及外部输入驱动的任意命令执行链路，不具备 XWorm 的核心远控能力。
- 原生程序仍包含开机启动注册表写入、全局按键轮询、进程枚举、按名称终止进程、屏幕捕获和子进程创建等启发式敏感行为。
- 本机 Defender 当前停用，无法在本机复现用户检测；最终需要外部复测或厂商样本分析。

## 总体方案

采用四层组合治理：

1. 补齐可执行文件身份和统一版本。
2. 收敛渲染器中不必要或过宽的敏感行为。
3. 建立可验证、可追溯的 Release 打包流程。
4. 准备误报申诉和可选代码签名流程。

每层单独实施和验证，不通过无关重构或删除核心功能来追求扫描结果。

## 版本与身份资源

- 本次下一正式版本按 `1.2.0` 设计，GitHub 标签为 `v1.2.0`。
- CMake 中建立唯一版本来源，并同时供 Qt 应用版本、关于窗口、更新检测和两个 Windows 版本资源使用。
- `appBlakholeUI.exe` 的产品名使用 `Blakhole UI`，文件说明使用“黑洞可视化配置工具”。
- `blackhole.exe` 的产品名使用 `Blakhole Renderer`，文件说明使用“Blakhole 本地渲染器”。
- 公司名或发布者文字使用项目作者标识 `XboxNahida`，不伪造公司或证书身份。
- 两个文件都写入文件版本、产品版本、原始文件名、版权和项目链接相关说明。
- 清单继续使用 `asInvoker`，不请求管理员权限。

## 自启动职责收敛

- 开机自启动只由 Qt UI 的 `autostart_registry` 模块管理，启动目标保持为 `appBlakholeUI.exe`。
- 原生渲染器不再直接写入或删除 `HKCU\Software\Microsoft\Windows\CurrentVersion\Run`。
- 旧 ImGui 配置页移除“开机自启”交互，但保留配置文件中的兼容字段读取，避免破坏旧配置格式。
- 从渲染器移除无调用的注册表写入实现，并在链接允许时移除不再需要的注册表导入。
- 迁移或删除前必须通过引用检查和配置兼容测试，确认 Qt 自启动功能不受影响。

## 进程生命周期收敛

- 删除启动时枚举并终止所有同名 `blackhole.exe` 的逻辑；程序不得终止不由当前实例创建的进程。
- 控制或监控模式使用命名互斥量限制重复控制实例；`--render` 子进程不使用全局互斥量，以允许多显示器渲染。
- 继续使用现有 Windows Job Object 管理多显示器子渲染器，并将监控模式创建的渲染进程纳入相同的“父进程退出即清理”语义。
- 停止时先向自己持有句柄的子进程发送正常关闭消息并限时等待；仍未退出时通过关闭带 `KILL_ON_JOB_CLOSE` 的 Job Object 清理。
- 原生渲染器不再调用 `OpenProcess(PROCESS_TERMINATE)` 或按进程名调用 `TerminateProcess`。
- Qt UI 继续通过自己的 `QProcess` 对象管理其启动的主渲染进程，不扫描系统中的同名进程。

## 全局热键收敛

- 将渲染循环中每帧调用 `GetAsyncKeyState` 检查 `Ctrl+Alt+R` 改为 Windows `RegisterHotKey` 和 `WM_HOTKEY` 消息。
- 只由主渲染实例注册热键；带 `--screen N` 的多屏子实例不注册，避免冲突。
- 热键注册失败时保持默认录制捕获状态并记录诊断，不影响渲染。
- 退出时注销热键，确保重启和多次启动不会遗留占用。
- Qt UI 已有的关闭渲染快捷键继续使用现有 `RegisterHotKey`，不改变用户设置。

## 屏幕捕获边界

- 保留 Windows Graphics Capture 和 DXGI 捕获，这是黑洞视觉效果的核心输入。
- 捕获帧只在本地内存和 GPU 中用于渲染，不新增落盘、上传、遥测或网络传输。
- 更新检查网络请求只存在于 Qt UI；`blackhole.exe` 保持没有网络库和下载接口。
- 发布前静态检查渲染器导入表，禁止新增 Winsock、WinHTTP、WinINet、URLDownload、远程线程和跨进程写内存 API。

## Release 构建卫生

- 构建目录保留未剥离 EXE，供崩溃转储和符号定位使用。
- 发布包中的 EXE 移除 DWARF 调试节区，不使用 UPX 或其他压缩壳。
- 打包脚本明确区分“可调试构建产物”和“发布文件”，分别计算 SHA-256。
- 发布包只包含运行必需的 EXE、DLL、QML、着色器、默认配置和许可证/说明文件。
- 生成中文 `SECURITY.md` 或发布安全说明，公开软件为何需要屏幕捕获、进程查询和自启动权限，并声明渲染器不联网。
- 每个正式 Release 记录版本号、Git 提交、两个 EXE 的 SHA-256、构建时间和测试结果。

## 误报申诉与签名

- 为 Microsoft 和实际命中的第三方杀软准备统一材料：检测名称、文件哈希、正式 Release URL、源码 URL、功能说明和无网络/无注入证据。
- 未经用户明确允许，不把 EXE 上传到 VirusTotal、杀软厂商或其他第三方服务。
- 代码签名作为推荐但非阻塞项；当前实施先完成可签名的稳定产物和签名脚本入口。
- 用户取得可信代码签名证书后，同时签署两个 EXE，并在打包后验证签名状态和时间戳。
- 签名不能替代行为加固和误报申诉，也不能保证所有引擎立即放行。

## 分阶段验证

### 阶段一：身份与构建卫生

- 先增加失败测试，验证两个 EXE 必须具有完整版本资源。
- 验证打包文件不含 DWARF 调试节区、不含 UPX 特征，仍保留 ASLR 和 DEP/NX。
- 验证唯一版本来源同时反映到 CMake、QML、更新比较和版本资源。

### 阶段二：自启动与进程管理

- 先增加静态和单元测试，证明旧代码仍包含渲染器注册表写入和按名称终止逻辑。
- 修改后验证 Qt 自启动开启、关闭、旧值迁移和重启状态不回归。
- 验证控制实例单例、多显示器多个 `--render` 实例、父进程退出清理和正常停止。
- 验证程序不会终止用户另行启动或路径不同的同名进程。

### 阶段三：热键与敏感导入

- 先建立热键契约测试，禁止渲染循环使用 `GetAsyncKeyState`。
- 验证 `Ctrl+Alt+R` 只触发一次、长按不重复、多屏不冲突、退出后可重新注册。
- 检查最终 `blackhole.exe` 导入表，确认无注册表写入、按名称终止、全局按键轮询、网络和注入类 API。

### 阶段四：外部复测

- 生成带版本、哈希和变更说明的候选 Release，不覆盖最后一个稳定发布包。
- 由至少一名曾持续报毒的用户复测下载、解压和运行阶段。
- 记录安全产品、引擎版本、检测名称、命中文件、SHA-256 和发生阶段。
- 若仍误报，使用相同候选文件提交厂商复核，不再通过随机改代码改变哈希碰运气。

## 回归范围

- 主程序与 Qt UI Release 全量构建。
- 开机自启动注册表测试。
- 桌面空闲、视频、游戏和三类名单检测。
- 始终显示、空闲显示和定时显示入口。
- 单屏、副屏、跨屏和一屏一黑洞模式。
- `Ctrl+Alt+R` 录制捕获切换与 Qt 关闭渲染快捷键。
- 主界面启动、停止、退出、崩溃清理和重复启动。
- 关于窗口、头像、更新检查和 Release 打包。
- UTF-8、空白检查和现有全部自动化测试。

## 风险与限制

- 移除按名称强制清理后，旧版本遗留的孤儿进程不会被新版本粗暴终止；Job Object 只保证新版本创建的子进程得到正确清理。
- 不同杀毒引擎可能继续因屏幕捕获或低下载量报警，这需要代码签名、信誉积累和厂商申诉共同改善。
- 每次构建都会改变哈希，申诉应针对最终候选文件，不能在申诉后重新编译再发布。
- 外部扫描服务会获得上传文件，必须由用户明确授权后使用。

## 不在本次范围

- 不关闭或绕过用户的杀毒软件、智能应用控制和系统安全策略。
- 不通过加壳、混淆、修改随机字节或伪造签名规避检测。
- 不删除屏幕捕获、媒体检测、游戏检测或多显示器核心功能。
- 不自动购买、申请或持有用户的代码签名证书。
