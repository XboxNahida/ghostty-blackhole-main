# 发布新版本指南

本文以 Windows x64 正式版 `1.2.2`、标签 `v1.2.2` 为例。发布附件名称固定为：

```text
BlakholeUI-v1.2.2-windows-x64.zip
```

## 1. 发布前准备

1. 确认唯一版本源 `cmake/AppVersion.cmake` 中的版本与目标标签一致。
2. 确认工作区没有来源不明的修改，并记录准备发布的 Git 提交。
3. 完成渲染器、Qt UI 和现有回归测试。不得为了降低误报删除屏幕捕获、媒体检测、游戏检测或多显示器功能。
4. 不使用 UPX、加壳、混淆、随机字节修改或伪造签名。没有可信证书时如实发布未签名版本。
5. 未经用户明确授权，不把候选 EXE 上传到 VirusTotal、杀毒厂商或其他外部扫描服务。

## 2. 构建与打包

打包必须从已提交且 tracked/staged 均干净的工作区开始。`release` 和构建目录属于忽略产物，不影响该门禁。提交最终源码后，在仓库根目录执行：

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File .\package_release.ps1
powershell -NoProfile -ExecutionPolicy Bypass -File .\tools\check_release_security.ps1
```

`package_release.ps1` 默认先在仓库内 `_build_v122_renderer` 和 `_build_v122_ui` 执行 `--clean-first` Release 构建，再保存原件哈希、复制文件并生成记录；构建目录尚未配置时，脚本会先在对应短目录运行 CMake 配置。它只对复制到 `release` 的两个 EXE 执行经 `--version` 验证的 GNU/MinGW `strip.exe --strip-debug`，不会剥离短目录中的构建原件。`-NoBuild` 只用于已有可信 clean build 的诊断，不得用于正式发布。

脚本不会清空 `release`，只覆盖自己管理的发布文件；因此压缩前必须人工检查该目录，确认没有个人文件、旧测试文件或其他不应公开的内容。哈希清单会递归记录当时 `release` 中的全部文件（仅排除清单自身），意外文件不会被隐藏。

打包成功后检查：

- `RELEASE_INFO.txt` 的版本、标签、提交、UTC/本地构建时间和 `Unsigned/NotSigned` 状态正确。
- `release_checksums.sha256` 使用小写路径、正斜杠和小写 SHA-256，完整覆盖解压目录；随包校验器报告 `RELEASE_CHECKSUMS_OK`，发布者安全检查报告 `RELEASE_SECURITY_OK version=1.2.2`。
- 发布 EXE 无 `.debug_*` 和 UPX 节，保留 ASLR、NX 与完整 VERSIONINFO。
- `blackhole.exe` 未出现发布检查脚本列出的常见网络、进程注入、注册表写入和 `GetAsyncKeyState` 导入。该检查不构成对所有实现形式的形式化证明。
- 构建原件哈希前后一致，且仍保留用于问题定位的调试节。

## 3. 生成候选压缩包

先在资源管理器或终端中复核 `release` 内容，再生成固定名称的 ZIP：

```powershell
$zip = ".\BlakholeUI-v1.2.2-windows-x64.zip"
$sidecar = "$zip.sha256.txt"
if (Test-Path -LiteralPath $zip) { Remove-Item -LiteralPath $zip -Force }
if (Test-Path -LiteralPath $sidecar) { Remove-Item -LiteralPath $sidecar -Force }
Compress-Archive -Path .\release\* -DestinationPath $zip -CompressionLevel Optimal
$hash = (Get-FileHash -Algorithm SHA256 -LiteralPath $zip).Hash.ToUpperInvariant()
Set-Content -LiteralPath $sidecar -Encoding ascii -NoNewline `
  -Value "$hash  BlakholeUI-v1.2.2-windows-x64.zip`r`n"

$sidecarLine = Get-Content -Raw -Encoding ascii -LiteralPath $sidecar
if ($sidecarLine -cne "$hash  BlakholeUI-v1.2.2-windows-x64.zip`r`n") {
    throw "ZIP SHA-256 sidecar verification failed"
}
```

记录 ZIP 的 SHA-256，并将 `BlakholeUI-v1.2.2-windows-x64.zip.sha256.txt` 与 ZIP 一同上传。ZIP 自身不能写入其内部的 `release_checksums.sha256`。最终候选包一旦用于复测或误报申诉，不要再重新压缩或编译。

## 4. GitHub Release 类型

- Draft：仅维护者可见的发布草稿，适合上传附件、检查文字和最终哈希，不会成为用户看到的正式最新版。
- Tag：指向某个 Git 提交的版本标记，本身不等于 Release。标签必须使用 `v1.2.2`，并准确指向 `RELEASE_INFO.txt` 记录的提交。
- Pre-release：公开的预发布版本，用于测试候选版。应用的正式版更新检查不会把它当作最新版。
- 正式 Release：公开且未勾选 Pre-release 的版本，会被 GitHub 的 latest Release 和应用更新检查识别。

## 5. 创建并发布

1. 在 GitHub 仓库的 Releases 页面选择“Draft a new release”。
2. 创建或选择标签 `v1.2.2`，确认目标提交与 `RELEASE_INFO.txt` 一致。
3. 标题填写清晰的中文版本名，正文列出主要变更、已知限制、安全边界和 ZIP 的 SHA-256。
4. 上传 `BlakholeUI-v1.2.2-windows-x64.zip` 和 `BlakholeUI-v1.2.2-windows-x64.zip.sha256.txt`。
5. 保持 Draft，重新下载附件并核对文件名、大小和 SHA-256。
6. 需要公开测试时勾选 Pre-release；准备正式发布时取消 Pre-release。
7. 发布正式 Release 后，从公开页面再次下载附件，在干净目录解压，运行 `verify_release_checksums.ps1` 和基本启动测试。下载用户不需要运行依赖构建目录和 `objdump` 的发布者安全检查。

## 6. 发布后验证

至少验证 Qt UI 启动、启动/停止渲染、单屏与多显示器、`Ctrl+Alt+R`、空闲/媒体/游戏检测、自启动开关和更新检查。记录测试系统、构建提交、ZIP SHA-256、两个 EXE SHA-256 以及结果。

屏幕捕获只在本地内存和 GPU 中用于渲染。`blackhole.exe` 不联网；Qt UI 只为版本检查访问 GitHub Releases，并通过系统浏览器打开下载页。文档和发布说明不得声称“绝对不报毒”。

## 7. 回滚与撤回

发现严重问题时，先把错误 Release 改回 Draft 或删除 Release 附件，避免继续下载；如标签指向错误提交，删除错误标签后在正确提交重新创建。不要静默替换同名附件，因为同名但不同哈希会破坏追溯和误报申诉证据。

回滚到上一稳定版时，在 Release 说明中明确受影响版本、停止分发原因和建议操作。修复后使用新版本号和新标签重新构建、验证和发布，不复用已公开版本的标签或哈希。

## 8. 误报申诉证据

收集并保存以下信息：检测名称、安全产品名称、引擎与规则版本、发生时间、命中文件、ZIP 和 EXE 的 SHA-256、正式 Release URL、源码 URL、构建提交、功能与敏感权限说明、`RELEASE_SECURITY_OK` 输出，以及渲染器未出现检查脚本所列常见网络、注入和注册表写入导入的结果。

只有在文件所有者明确同意后，才向对应厂商提交最终候选样本。申诉必须针对已发布且哈希固定的文件；不得通过反复改代码、重新压缩或伪造签名来碰撞不同检测结果。
