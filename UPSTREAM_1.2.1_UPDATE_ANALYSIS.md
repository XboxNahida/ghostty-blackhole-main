## 1. 结论先行

### 1.1 不应把 `update-1.2.1/` 覆盖或合并到根目录

作者的 `v1.2.1` Release 指向提交：

```text
1c570a80cda8caea1ff27b7554123833e4b27e9e
```

本地 `update-1.2.1/` 与该提交的源码树逐文件一致；对排序后的“相对路径 + 文件 SHA-256”清单再求摘要，两边均为：

```text
c89ebbba5d7a1810c534264d01ca5407cd557aa744673c8169473b62f0e0ccaf
```

当前 Ubuntu 分支与 `main` 的共同祖先同样是 `1c570a8`：

```text
git merge-base main port/ubuntu-wayland
1c570a80cda8caea1ff27b7554123833e4b27e9e
```

因此，当前 Ubuntu 工程不是“基于一个早于 v1.2.1 的源码再等待合并”，而是已经从 v1.2.1 对应的完整源码提交上开始移植。根目录与更新包的差异主要是后续 Ubuntu/Linux 平台实现，不是缺少作者更新。直接覆盖会删除或倒退：

- Linux CMake 目标与安装/CPack 规则；
- GNOME Shell 扩展与 D-Bus 控制面；
- GNOME IdleMonitor、MPRIS、XDG autostart；
- Linux 渲染器、Portal/PipeWire 和显示器选择代码；
- Linux 平台测试与 Ubuntu 文档。

### 1.2 所谓“1.2.0 → 1.2.1”没有一个可直接复现的官方 Git 差分

调查时上游公开 Release/标签链只有 `v1.1` 和 `v1.2.1`，没有可作为左端点的公开 `v1.2.0` 标签。更关键的是，`v1.2.1` 源码包内部仍统一声明自己是 `1.2.0`：

- `cmake/AppVersion.cmake`：`BLACKHOLE_VERSION "1.2.0"`；
- `package_release.ps1`：写入 `Version: 1.2.0`、`Tag: v1.2.0`；
- `tools/check_release_identity.ps1`：强制校验 `1.2.0`；
- `tools/check_release_security.ps1`：强制校验 `1.2.0`；
- `SECURITY.md`：描述的也是 `v1.2.0` 构建身份。

这说明作者发布的 `v1.2.1` 标签/标题与包内二进制及构建身份没有同步。不能把发布说明九条内容武断地当成一个严格的 `v1.2.0..v1.2.1` Git diff。

### 1.3 最合理的两种阅读口径

1. **发布口径**：作者的九条“主要更新”是对从旧公开版到 `v1.2.1` 的累计功能概述。
2. **内部检查点口径**：提交 `10a9e13` 的标题是“合并 v1.2.0 功能与发布改进”。若把它临时视为内部 1.2.0 检查点，则 `10a9e13..1c570a8` 的真实新增主要只有：
   - 修复 Release 构建中鼠标跟随坐标失效；
   - 新增黑洞出生位置、自由移动速度和独立旋转速度链路；
   - 支持两个本地收款码资源的并排打包展示；
   - 新增渲染器启动握手、诊断状态机和错误弹窗；
   - 修复失败进程生命周期、日志跨快照/截断读取和退出前日志消费。

该内部检查点之后的统计为 27 个文件、约 2943 行新增、78 行删除；其中包含大量设计/实施文档和测试。发布说明中的媒体/游戏检测、名单、更新提醒、安全发布、预览背景等功能，大多在 `10a9e13` 之前已经进入源码。因此发布说明是累计摘要，而不是上述检查点后的精确变更清单。

### 1.4 2026-07-14 最终产品范围决定

本项目不是跨平台 fork，也不以完整复刻 Windows 版为目标。最终定位是：

> **只服务当前这台 Ubuntu 26.04、GNOME 50、Wayland、NVIDIA RTX 3060、单显示器主机的本地黑洞桌面效果。**

由此确定以下约束：

- 不维护、不验证 Windows 构建与行为；后续 Linux 改动破坏 Windows 兼容性不构成阻塞。
- 不继续追随上游功能，不把作者未来更新当作本项目必须合并的版本线。
- 不在本轮新增产品功能，也不改进黑洞视觉算法；只修复本机核心链路中的真实缺陷。
- GNOME Shell 50 扩展 + 本地 D-Bus 是唯一正式运行路径。
- 只支持当前单显示器；删除或隐藏主屏/副屏/跨屏/一屏一黑洞相关入口，不做多屏实现或测试。
- Windows 专属能力若没有简单、稳定且符合 GNOME/Wayland 模型的替代，直接放弃；非核心功能不为了“对齐”而增加复杂度。
- 可以保留未参与 Linux 构建的 Windows 历史源码作为参考，但不为其投入兼容、清理或测试成本。

## 2. 调查方法与证据边界

本次使用了以下证据，不依赖文件夹名称作推断：

1. GitHub Release 页面确认 `v1.2.1` 指向 `1c570a8`。
2. 将 `git archive 1c570a8` 解压到临时目录，与 `update-1.2.1/` 执行递归逐文件比较，结果无差异。
3. 对两棵树的相对路径及文件哈希清单做二次 SHA-256，摘要完全一致。
4. 用 `git merge-base` 确认 Ubuntu 分支确实从 `1c570a8` 分叉。
5. 阅读关键 C++、QML、PowerShell、CMake、测试与安全文档，并将发布说明逐项映射到具体实现。
6. 对现有 Ubuntu 构建目录执行 13 个 CTest 测试；12 个直接通过，`application_catalog_tests` 因当前命令环境不能连接 Wayland/X11 而失败；设置 `QT_QPA_PLATFORM=offscreen` 后该测试单独通过，随后同一环境下整套测试 13/13 通过。由此可判断首次失败是无显示环境问题，不是业务断言失败。

## 3. 作者更新的实际功能细节

### 3.1 渲染器启动错误弹窗

核心不只是一个 QML 弹窗，而是一套启动协议和有界诊断状态机：

- 状态：`Stopped / Starting / Ready / Stopping / Failed`；
- 失败类型：缺少文件、进程无法启动、Ready 前退出、初始化失败、8 秒 Ready 超时；
- 渲染器日志用 `[OK] Ready, entering main loop` 表示就绪，用 `[FAIL]` 表示初始化失败；
- 启动前先验证 EXE、顶点 shader、片段 header 和 `blackhole.glsl`；
- 日志读取能处理追加、文件替换、截断、不完整行和跨快照 marker；
- 弹窗展示摘要、技术详情、日志路径，并支持复制详情、打开日志目录；
- 展示内容上限约 8 KiB，启动日志总排空读取也有上限，避免无界内存/磁盘读取。

主要文件：

- `Blakhole_UI/core/renderer_startup_diagnostics.{h,cpp}`
- `Blakhole_UI/components/RendererErrorDialog.qml`
- `Blakhole_UI/core/blackholecore.cpp`
- `tests/renderer_startup_diagnostics_tests.cpp`

### 3.2 视频播放与游戏检测

Windows 实现不是简单进程名黑名单，而是分层判定：

1. 排除桌面、系统壳层和黑洞自身窗口，避免自我误判。
2. 读取 `SHQueryUserNotificationState`，识别 D3D 独占全屏/演示模式。
3. 判断当前显示器上的无边框全屏应用。
4. 用 Windows/Steam 已知游戏路径做保守游戏识别；Steam 重点匹配 `steamapps/common`。
5. 对专用播放器、浏览器和 UWP 媒体窗口分类；浏览器还要求标题出现视频关键词，减少普通网页误报。
6. 优先读取 Windows Global System Media Transport Controls 的播放状态，并核对来源应用；必要时才回退音频峰值。
7. 把检测原因暴露到 UI，便于判断为何没有启动。

这比只看“有声音”可靠，因为暂停媒体、普通浏览器标签、系统桌面和自身渲染器不会轻易被当作正在播放的视频。

主要文件：

- `src/foreground_window.{h,cpp}`
- `src/game_detection.{h,cpp}`
- `src/media_session.{h,cpp}`
- `Blakhole_UI/core/blackholecore.cpp`

### 3.3 应用名单添加方式

源码中共有三种名单：

- 始终允许触发名单（白名单）；
- 媒体应用提示名单；
- 前台强制不触发名单。

每个名单都提供三种添加入口：

1. 手工输入进程名；
2. 从当前运行应用列表中选择；
3. 选择本地可执行文件并提取进程名、显示名和图标。

条目会去空白、去重并持久化到 `blackhole_idlelist.txt`。运行应用目录会排除自身进程和部分系统目录进程，并按显示名排序。对应文件是 `IdleListPanel.qml`、`RunningApplicationPicker.qml`、`IdleListConfig.qml` 和 `application_catalog.{h,cpp}`。

### 3.4 开机自启动与空闲模式修复

Windows 端由 Qt UI 直接读写当前用户的 `HKCU\Software\Microsoft\Windows\CurrentVersion\Run`，开关切换后立即同步并返回具体错误码，而不是只把布尔值写进配置文件。

空闲模式方面修复了 UI 入口与监控进程启动链路，并把媒体名单触发崩溃、桌面/自身误判、全屏检测分歧一并收敛。主 UI 的“应用并启动”会根据“始终显示/空闲检测”决定直接启动渲染器还是启动监控逻辑。

### 3.5 关于页面、自定义头像与收款码

- “关于”从独立窗口改为主窗口内页面，并修复重复进入/返回问题；
- 头像支持 PNG、JPEG、WEBP，读取后会重新编码到应用数据目录，不直接依赖原图片路径；
- 非法格式、读取失败、目录创建失败和写入失败都有明确状态；
- 两个收款码使用 `QR_payment.jpg` 和 `WeChat_QR.png`，只有两者都存在时才展示；
- 收款码不进入公开源码，由本地打包阶段注入，避免把个人支付资源提交到公开仓库。

因此“源码中没有收款码图片”不是缺文件，而是作者刻意设计的本地发布资源边界。

### 3.6 GitHub Release 更新提醒

- UI 启动后异步请求官方仓库 `releases/latest`；
- 只接受非 draft、非 prerelease 的正式 Release；
- 版本采用语义化数字比较，不做简单字符串比较；
- 只允许 HTTPS，API 响应最终地址必须仍在 `api.github.com`；
- 下载页必须是 `github.com` 的 Release 地址；
- 响应上限 1 MiB，并有 8 秒超时；
- 支持自动检查、手动检查、更新红点、查看说明、忽略当前版本；
- 程序只打开系统浏览器，不在应用内下载、解压或替换二进制。

### 3.7 进程管理、系统热键与发布安全

- 多渲染器使用 Windows Job Object，父进程退出时由 `KILL_ON_JOB_CLOSE` 清理受管子进程；
- “一屏一黑洞”为每个显示器启动独立渲染进程，单个进程负责一个屏幕；
- 录制捕获切换改用 `RegisterHotKey` 注册 `Ctrl+Alt+R`，不再轮询全局键盘状态；
- 主 UI 的关闭黑洞快捷键也使用系统热键注册，并支持自定义组合；
- 发布包生成 `RELEASE_INFO.txt`、递归 SHA-256 清单和独立校验脚本；
- 发布前检查常见网络下载、进程注入、远程控制相关导入；
- 明确普通用户权限、无管理员请求、无 UPX/加壳/混淆、无伪造签名；
- 安全说明同时承认屏幕捕获、进程检测和全局热键仍可能触发启发式误报。

安全边界需要精确表述：**Windows 渲染器没有网络功能，但 Qt 配置 UI 有只读 GitHub Release 检查网络功能**。不能笼统宣称整个产品完全不联网。

### 3.8 黑洞预览和星空背景

- 预览 shader 被明确纳入 Qt 资源和发布包；
- `Starry_sky_background.png` 同时支持 qrc、开发目录和发布目录查找；
- 缺失时仍有生成背景回退，不应导致 UI 直接崩溃；
- 修复了预览 shader/背景在源码可用但 Release 漏打包的问题。

### 3.9 多显示器与录制捕获

Windows 支持四种显示方式：主屏、副屏、跨屏和一屏一黑洞。跨屏会分别捕获主副屏并拼入虚拟桌面坐标；一屏一黑洞用多进程隔离每屏渲染。

“允许录制捕获”不是继续实时递归捕获桌面：开启时让录屏软件能够捕获黑洞窗口，同时冻结黑洞所使用的桌面纹理；关闭时重新排除黑洞窗口并恢复实时桌面纹理更新。`Ctrl+Alt+R` 可在运行时切换。这是在“录得到黑洞”与“避免捕获反馈回路”之间做的取舍。

### 3.10 发布说明没有点明、但内部检查点后确实新增的细节

- 黑洞出现位置由固定随机开关扩展为可配置出生位置；
- 自由移动速度与吸积盘旋转速度拆分，避免一个速度同时改变两种运动；
- 修复 Release 模式鼠标跟随坐标失效；
- 本地收款资源改为双图并排布局；
- 启动失败后锁存自动重试，用户手动重试时才清除失败状态，避免空闲监控不断拉起一个必然失败的渲染器。

## 4. 按本机化目标重新裁定遗留功能

| 能力 | 最终处置 | 理由 |
|---|---|---|
| GNOME Shell 50 合成器效果 | **唯一正式渲染路径，保留** | 这是本机桌面实时黑洞效果的核心。 |
| Qt 配置 UI | **保留并瘦身** | 只保留确实能控制 GNOME 扩展的设置。 |
| 黑洞 shader、预设、预览 | **冻结算法，保留** | 不继续改进视觉算法，只修复导致本机不可用的兼容性错误。 |
| 启动/停止与 D-Bus 错误弹窗 | **必须修复** | 是核心控制链路，当前 GNOME 启动错误没有接入详细弹窗。 |
| GNOME 空闲检测与活动恢复 | **必须保留** | 属于屏保式体验的核心；当前已基本落地。 |
| MPRIS 媒体抑制 | **保留现状，不扩展名单体系** | Linux 已有自然替代且代码已工作；只验证本机常用播放器。 |
| XDG 用户自启动 | **保留** | 本机日常使用需要，且已有简单稳定实现。 |
| 版本身份 | **修正为本机 Ubuntu 变体** | 避免文档、包名和 About 信息继续错误声称 1.2.0。 |
| GitHub Release 更新检查 | **Linux 构建停用或移除** | 本项目不再追随上游，作者 Release 又是 Windows 产物；无需建立新的联网更新通道。 |
| Windows 游戏/前台窗口检测 | **放弃** | Wayland 下没有简单可靠的等价方案，也不是黑洞效果的核心。 |
| 三类应用名单与运行应用选择器 | **Linux UI 移除** | 当前 Linux 空闲路径不消费这些名单；不再为对齐 Windows 编写 `/proc` 枚举或前台检测。 |
| 全局关闭热键 | **保留，交由 GNOME Shell 扩展实现** | GNOME 50 提供原生 Wayland 全局快捷键；无需原始输入捕获或 JNI/JNA。 |
| 录制捕获与 `Ctrl+Alt+R` | **Linux UI 移除** | 属于 Windows WGC 窗口捕获语义，GNOME 合成器路径不需要复刻。 |
| 多显示器与显示器选择 | **代码不再演进，Linux UI 移除** | 本机只有单显示器，不实现、不测试、不保留产品入口。 |
| Portal/PipeWire 全屏 renderer | **退出正式产品路径** | 当前 GNOME 合成器已经解决桌面实时效果；旧路径只作历史代码，不再回归或扩展。 |
| 独立 `blackhole-renderer` 进程诊断 | **不作为产品要求** | GNOME 扩展是唯一运行路径；只保留不会妨碍构建的历史实现。 |
| About/自定义头像 | **可保留，不再扩展** | 已跨平台可用且不干扰核心；不为它安排新功能。 |
| 双收款码 | **不纳入本机化任务** | 与核心功能无关，也没有必要建立私有资源注入流程。 |
| Windows/PowerShell 发布安全检查 | **不迁移** | 本项目只需要本地 Linux 构建和安装验证。 |

## 5. 当前真正需要解决的问题

### P0：GNOME 主路径错误信息断链

D-Bus 服务不存在、扩展未启用或 `Start` 调用失败时，当前代码只发 `rendererError`，而 QML 详细弹窗监听 `rendererStartupFailed`。应把 GNOME 错误直接接入现有弹窗；不需要复刻 Windows 渲染器的全部日志协议。

### P0：Linux UI 含有无效或无关入口

多屏、运行应用名单、录制捕获、Windows 捕获方式会造成“保存了但不生效”的假象，应删除或隐藏。全局停止热键是例外：它已经由 GNOME Shell 扩展提供真实后端，继续保留现有 UI。

### P1：本机版本身份和文档不准确

程序仍自报 `1.2.0`，README 仍把项目描述为双平台产品并称“1.2.1 尚未合并”。应改为本机 Ubuntu 变体身份，同时停用 Linux 的上游在线更新检查。

### P1：GNOME 扩展实际读取的设置少于 UI 提供的设置

扩展目前明确读取预设、切换间隔、播放模式、固定大小、洞半径、移动速度和旋转速度。其他设置只有在证明会影响 GNOME 50 主路径时才应继续显示；不为保存格式中存在但主路径不消费的参数补功能。

### P2：历史迁移文档和旧 renderer 容易制造错误目标

Portal、多屏、跨平台和独立 renderer 已经完成过探索，但不再是当前产品方向。历史代码可以保留，文档必须明确它们不属于验收范围。

## 6. Ubuntu GNOME 50 单屏本机化收尾计划
